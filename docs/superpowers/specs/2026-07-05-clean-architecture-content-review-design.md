# クリーンアーキテクチャ教材 コンテンツレビュー 設計

- 日付: 2026-07-05
- 対象リポジトリ: `clean-architecture`（`docs/` 配下の全9章）

## 背景・目的

本リポジトリは「クリーンアーキテクチャ完全ガイド」という日本語の教材（全9章・約50ファイル）。
これまで内容の正確性・章間の一貫性・読みやすさについて体系的なレビューが行われておらず、
すでに以下のような問題が確認できている。

- 誤字・簡体字混入: `docs/01-introduction/01-overview.md` の「変更对応」（"対"が中国語簡体字）、
  `COMPLETION-REPORT.md` の「団隊」「理動」など
- ドキュメント間の数値矛盾: `README.md` は「合計50ファイル、15,000行以上」、`COMPLETION-REPORT.md` は
  「合計25ファイル、30,000行以上」と記載されており、両者が食い違っている

これらを一次情報源（Uncle Bob のブログ、書籍『Clean Architecture』）と照合しながら解消し、
教材全体の品質を引き上げる。

## 一次情報源（正確性の判定基準）

- **主**: Uncle Bob（Robert C. Martin）のブログ（blog.cleancoder.com）の関連記事
  - "The Clean Architecture"（4重円モデル、依存性のルールの原典）
  - "The Dependency Rule"、"Screaming Architecture" など関連記事
- **補助**: 書籍『Clean Architecture: A Craftsman's Guide to Software Structure and Design』の一般に
  知られている内容・構成（層区分、用語、原則の説明）。Web上で参照できない詳細部分の照合に使う。

レビュー着手前に、ブログの主要記事を確認し、照合用の要点メモ（層構造、依存性のルール、
用語の正確な英語表記など）を作成してから各章のレビューに入る。

## レビュー範囲

`docs/` 配下の全9章（`01-introduction` 〜 `09-tools-and-resources`）、各章のREADME・詳細ファイルすべて。

**対象外**:
- `ebook-output/`（`docs/` から自動生成されるビルド成果物。ソース修正後、再ビルドはユーザーが別途実行）
- 現在ワーキングツリーに存在する、diagram の mermaid 化に関する未コミットの変更（本レビューとは無関係の
  作業中の変更のため、触れない）

## レビュー観点

各ファイルに対して以下3つの観点を適用する。

1. **正確性**
   - Clean Architecture の概念説明が一次情報源と矛盾していないか
   - 参考書籍の著者・出版社・出版年などの事実情報が正しいか
   - コード例（TypeScript/Java/Python 等）が技術的に妥当か
2. **一貫性**
   - 章をまたいだ用語・層名・命名規則の統一
   - ファイル間の統計値（ファイル数・行数・学習時間など）の食い違い
   - 章内・章間の相互リンクが有効であること
3. **読みやすさ**
   - 誤字脱字、簡体字などの混入
   - 冗長な装飾（過剰なASCIIアート・絵文字）の簡潔化
   - 文章として分かりやすいか（曖昧な表現、日本語として不自然な箇所）

### 層用語（4層モデル）の扱い

本教材は実務でよく使われる Presentation/Application/Domain/Infrastructure の4層モデルを採用しているが、
Uncle Bob の原典は Entities/Use Cases/Interface Adapters/Frameworks & Drivers の4重円である。

この不一致は「誤り」として全面的に書き換えるのではなく、**実務的な変形であることを明記した上で維持**する。
`01-introduction`（導入章）で一度、4層モデルと原典の4重円との対応関係を明確に説明する記述を追加・調整し、
以降の章では現行の4層命名をそのまま使用する。

## 進め方

1. **事前準備**: Uncle Bob のブログから主要記事を確認し、照合用の要点メモを作成する
2. **章ごとに順次レビュー→即修正**: `01-introduction` から `09-tools-and-resources` まで、
   1章ずつ全ファイルを読み、3つの観点でチェックした上でその場で修正する
   - サブエージェントによる並列化は行わず、単一セッションで一貫した基準を保ちながら進める
3. **章完了時の報告**: 各章が終わるごとに、チャットで簡潔に「見つけた問題→修正内容」を報告する
   （章ごとの独立したレビューログファイルは作らない）
4. **横断的な整合性の解消**: 全9章の修正が終わった後、`README.md` と `COMPLETION-REPORT.md` の
   ファイル数・行数など、章をまたぐ統計の食い違いをまとめて解消する
5. **コミット**: 本レビュー作業によるコミットは、ユーザーの明示的な指示があった場合のみ行う
   （既存の未コミット変更とは切り分けて、レビューによる変更のみをステージする）

- 大規模な構成変更（章の統廃合、章立ての変更）は本レビューのスコープ外。文章単位の修正に限定する
